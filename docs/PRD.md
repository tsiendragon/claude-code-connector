## 1. 这个工具解决什么问题

### 1.1 先理解 Claude Code 是什么

Claude Code（以下简称 "claude CLI"）是 Anthropic 提供的 AI 编程助手，但它**不只是聊天**——它是一个能在你电脑上真正干活的 AI 员工：

- 自己读代码、改文件、跑测试
- 发现 bug 后自动修复并验证结果
- 执行终端命令、分析日志
- 跨多个文件完成重构任务

你可以把它理解成：**一个坐在你电脑前、能自主操作的 AI 程序员**。

### 1.2 现在的问题：只能手动盯着它

目前 Claude Code 只能在**终端窗口**里使用——你打开终端、输入问题、等它回答、再输入下一个指令。

这意味着：

- 你不能让它**在后台自动跑**，它需要你一直在旁边
- 你不能同时开 10 个任务让它**并行工作**
- 你不能把它接入**自动化流程**（比如每次提交代码后自动触发）
- 你一旦关掉终端或重启电脑，之前的工作上下文就**消失了**
- 如果它中途需要你确认某个操作，没人在旁边的话整个任务就**永远卡住**

### 1.3 ccc 做什么

ccc 是一个**遥控器**，让你的程序（Python 脚本、自动化平台、其他 AI）能够远程控制 Claude Code：

- 发送任务指令给它
- 知道它什么时候做完了
- 读取它的工作结果
- 在它需要确认时自动替它按"同意"
- 同时管理多个并行工作的 Claude Code 实例

> **为什么不直接调用 AI 的 API？**
>
> 直接调 API 只能得到文字回答，就像发邮件问 AI 一个问题——AI 告诉你"你应该把第 42 行改成这样"，但它不会真的去改。
>
> Claude Code 会**真的去改那一行代码**、跑测试、如果测试挂了再回头修。这是两件完全不同的事。

## 2. 真实使用场景

### 场景一：代码提交后自动审查

**背景**：团队每天有几十个 PR，人工 code review 耗时且容易漏看安全问题。

**没有 ccc 的世界**：只能手动打开终端，粘贴代码，等 Claude Code 看完，再把结果复制到 PR 评论里。完全无法规模化。

**有了 ccc**：每次有人提交代码，CI 系统自动启动一个 Claude Code 实例，让它审查这次的改动，审查完后把结果自动写入 PR 评论，全程无人值守。

```
开发者提交 PR
    → CI 触发 ccc 脚本
    → Claude Code 分析代码改动
    → 自动在 PR 里留下 review 评论
    → 整个过程：0 人工干预
```

---

### 场景二：多个项目同时并行处理

**背景**：公司有 20 个微服务仓库，需要把所有仓库的日志格式统一升级。

**没有 ccc 的世界**：一个一个来，每个仓库手动开终端、让 Claude Code 改、确认、关掉，再开下一个。20 个仓库得花一整天。

**有了 ccc**：一个脚本同时启动 20 个 Claude Code 实例，分别处理 20 个仓库，并行跑完，20 分钟搞定。每个实例有自己独立的上下文，互不干扰。

```
Python 脚本
    → 同时创建 20 个 Claude Code 会话
    → 每个会话处理一个仓库
    → 全部完成后汇总结果
    → 耗时：原来的 1/20
```

---

### 场景三：AI 指挥 AI（多智能体协作）

**背景**：你想让一个"总指挥 AI"把大任务拆解，分配给多个"执行 AI"并行完成，最后汇总结果。

**没有 ccc 的世界**：总指挥 AI 只能生成文字计划，无法真正驱动其他 AI 去执行。

**有了 ccc**：总指挥 AI（任何 AI 程序）可以通过 ccc 创建并控制多个 Claude Code 实例，把子任务分配下去，等待结果，再基于结果继续决策。

```
总指挥 AI
    ├── 通过 ccc 创建"写功能"实例 → 实现新功能
    ├── 通过 ccc 创建"写测试"实例 → 编写测试用例
    └── 通过 ccc 创建"写文档"实例 → 更新文档
         ↓ 全部完成
    总指挥汇总，发现测试失败，让"写功能"实例修复
```

---

### 场景四：无人值守的夜间任务

**背景**：想让 AI 在下班后自动处理积压的技术债务（修 lint 警告、补充注释、更新依赖）。

**没有 ccc 的世界**：没法"让它自己跑"——Claude Code 中途可能需要确认"是否允许执行这个命令"，没人在旁边按确认键，任务就永远卡在那里。

**有了 ccc**：脚本设定好任务，ccc 在后台监控 Claude Code 的状态。如果它需要确认权限，ccc 自动替它按"允许"；如果任务做完了，ccc 记录结果并启动下一个任务。你第二天来看结果就行。

---

### 场景五：平台重启不丢失进度

**背景**：一个长期运行的 AI 任务平台，需要在服务重启后恢复所有正在进行的任务。

**没有 ccc 的世界**：平台重启后不知道哪些 Claude Code 还活着、任务进行到哪了，只能全部重来。

**有了 ccc**：ccc 持久化记录每个会话的状态，平台重启后直接重新连接到还在运行的 Claude Code 实例，从中断处继续，不丢失上下文。

## 4. 核心能力要求

| 能力 | 说明 |
|---|---|
| **发送任务并等待完成** | 把指令发给 Claude Code，准确判断它什么时候真正做完（不能过早、不能超时） |
| **读取工作结果** | 从 Claude Code 的输出里提取干净的结果文本 |
| **自动响应确认弹窗** | 检测到"是否允许执行"等提示时，按照预设策略自动响应，不让任务卡死 |
| **并发管理多个实例** | 同时管理 10+ 个 Claude Code 会话，每个独立、互不干扰 |
| **会话持久化** | 记录每个会话状态，程序重启后能重新连接 |

### 成功指标

| 指标 | 目标 |
|---|---|
| 判断 Claude Code 完成任务的准确性误差 | ≤ 500ms |
| 程序重启后重连已有会话的成功率 | ≥ 99% |
| 最大并发会话数 | ≥ 10 |

### 不做的事

- 不支持 Windows（底层依赖 Linux/macOS 的 tmux）
- 不是 Anthropic API 的替代品（两者解决不同的问题）
- 不做用户权限管理，不面向公网部署
